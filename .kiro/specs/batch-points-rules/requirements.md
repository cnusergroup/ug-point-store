# 批量发分积分规则

## 背景
当前批量发分时，Admin 手动输入每人积分数值，没有标准化的积分规则。需要实现可配置的积分规则，让 SuperAdmin 设置各角色的标准积分值，Admin 发分时自动适配，不可手动修改。

## 积分规则

### (一) 活动组织积分（固定分配）
- **UGL (UserGroupLeader)**: 默认 50 分/人/场
- **志愿者 (Volunteer)**: 默认 30 分/人/场，每场最多 N 位（默认10，可配置）

### (二) Speaker 积分（AWS技术浓度分类）
- **A类（100% AWS内容）**: 默认 100 分
- **B类（外部技术，关联性低）**: 默认 50 分
- **圆桌嘉宾**: 默认 50 分

## 功能需求

### REQ-1: SuperAdmin 积分规则配置
- SuperAdmin 在功能设置页面新增「积分规则配置」区域
- 可配置字段：
  - UGL 每场积分（正整数）
  - 志愿者每场积分（正整数）
  - 志愿者每场最大人数（正整数，默认10）
  - Speaker A类积分（正整数）
  - Speaker B类积分（正整数）
  - 圆桌嘉宾积分（正整数）
- 配置存储在现有 Settings 表（feature-toggles）中

### REQ-2: Admin 批量发分流程改造
- 现有流程：选角色 → 选活动 → 选人 → 手动填积分 → 提交
- 新流程：选活动 → 选角色 → （Speaker 时选类型）→ 选人 → 积分自动填入（只读）→ 提交
- 流程步骤：
  1. 先选择关联活动
  2. 选择目标角色（Leader / Speaker / Volunteer）
  3. 如果选 Speaker，额外显示类型选择器：A类 / B类 / 圆桌嘉宾
  4. 选择发分用户
  5. 积分值根据角色+类型自动从配置读取，显示为只读（不可手动修改）
  6. 提交发分

### REQ-3: 志愿者人数限制
- 选择 Volunteer 角色时，选中人数不能超过配置的最大人数
- 超过时显示提示信息："每场活动最多选择 {maxCount} 位志愿者"
- 前端阻止提交，后端也做校验

### REQ-4: 后端校验
- 后端接收批量发分请求时，验证积分值是否与配置一致
- 如果前端传来的积分值与配置不匹配，拒绝请求
- 志愿者人数超限时拒绝请求

### REQ-5: 同一活动同一身份防重复发分
- 同一场活动 + 同一用户 + 同一角色（targetRole），只能获取一次积分
- 不同角色可以获取多次（如用户既是 Leader 又是 Speaker，同一活动可以分别获得 Leader 积分和 Speaker 积分）
- 后端在发分前查询 BatchDistributions 表，检查该活动+角色组合是否已有该用户的发分记录
- 如果有重复，返回错误并列出已获得积分的用户
- 前端在选人列表中标记已获得该角色积分的用户（灰色/不可选），提示"该用户已在此活动中以此身份获得积分"
