# Requirements Document

## Introduction

Add a new `OrderAdmin` role to the Points Mall system. OrderAdmin is a standalone, exclusive role dedicated to order management. Users with this role can only access order-related admin APIs and the orders management page. The role cannot coexist with other roles (Admin, SuperAdmin, Speaker, UserGroupLeader, Volunteer). SuperAdmin is the sole authority for creating and managing OrderAdmin users.

## Glossary

- **System**: The Points Mall application (frontend + backend)
- **OrderAdmin**: A new exclusive user role limited to order management operations
- **SuperAdmin**: The highest-privilege administrator role that can manage all users and settings
- **Admin**: A general administrator role with broad admin panel access
- **Role_Selector**: A UI component that allows selecting one or more roles for invite generation or role assignment
- **Invite_Link**: A registration URL generated by SuperAdmin that assigns specific roles to the registering user
- **Orders_Page**: The admin orders management page at `/pages/admin/orders`
- **Admin_Dashboard**: The admin panel index page at `/pages/admin/index`
- **Settings_Page**: The user settings page at `/pages/settings/index`
- **Navigation**: The set of menu entries visible to a user based on their role
- **Order_API**: The set of admin order endpoints: `GET /api/admin/orders`, `GET /api/admin/orders/stats`, `PATCH /api/admin/orders/:id/shipping`
- **Feature_Toggle**: A server-side configuration flag that controls access to specific admin features
- **Locale_File**: A frontend translation file containing i18n key-value pairs for a specific language

## Requirements

### Requirement 1: Add OrderAdmin to UserRole Type

**User Story:** As a developer, I want OrderAdmin defined as a valid UserRole, so that the type system enforces correct role handling across the codebase.

#### Acceptance Criteria

1. THE System SHALL include `OrderAdmin` as a valid value in the `UserRole` type definition in `packages/shared/src/types.ts`
2. THE System SHALL NOT include `OrderAdmin` in the `ADMIN_ROLES` array
3. THE System SHALL NOT include `OrderAdmin` in the `REGULAR_ROLES` array
4. THE System SHALL include `OrderAdmin` in the `ALL_ROLES` array
5. THE System SHALL define a new constant `EXCLUSIVE_ROLES` containing `OrderAdmin` for role exclusivity checks

### Requirement 2: Invite System for OrderAdmin

**User Story:** As a SuperAdmin, I want to generate invite links with the OrderAdmin role, so that I can onboard dedicated order management staff.

#### Acceptance Criteria

1. WHEN a SuperAdmin opens the invite generation form, THE Role_Selector SHALL display `OrderAdmin` as an available role option
2. WHEN a non-SuperAdmin Admin opens the invite generation form, THE Role_Selector SHALL NOT display `OrderAdmin` as an available role option
3. WHEN `OrderAdmin` is selected in the Role_Selector, THE System SHALL deselect all other roles and prevent selecting additional roles
4. WHEN any non-OrderAdmin role is selected after OrderAdmin was selected, THE System SHALL deselect `OrderAdmin` before selecting the new role
5. WHEN an invite is generated with the `OrderAdmin` role, THE System SHALL create the invite with `roles: ['OrderAdmin']` only

### Requirement 3: Backend Access Control for OrderAdmin

**User Story:** As a system operator, I want OrderAdmin users restricted to order-related APIs only, so that they cannot access or modify other admin resources.

#### Acceptance Criteria

1. WHEN an OrderAdmin user sends a request to `GET /api/admin/orders`, THE System SHALL process the request and return the orders list
2. WHEN an OrderAdmin user sends a request to `GET /api/admin/orders/stats`, THE System SHALL process the request and return order statistics
3. WHEN an OrderAdmin user sends a request to `PATCH /api/admin/orders/:id/shipping`, THE System SHALL process the request and update the shipping status
4. WHEN an OrderAdmin user sends a request to any admin API outside the Order_API set, THE System SHALL return HTTP 403 FORBIDDEN
5. THE System SHALL allow OrderAdmin users to bypass the `adminOrdersEnabled` Feature_Toggle check (OrderAdmin always has order access)
6. WHEN an OrderAdmin user sends a request to `GET /api/admin/orders/:id`, THE System SHALL process the request and return the order detail

### Requirement 4: Frontend Login Redirect for OrderAdmin

**User Story:** As an OrderAdmin user, I want to be redirected to the orders page after login, so that I can start managing orders immediately.

#### Acceptance Criteria

1. WHEN an OrderAdmin user completes login, THE System SHALL redirect the user to `/pages/admin/orders` instead of the default hub page
2. WHEN a user has both OrderAdmin and other roles (edge case from data migration), THE System SHALL treat OrderAdmin as the primary role and redirect to `/pages/admin/orders`

### Requirement 5: Frontend Orders Page Access for OrderAdmin

**User Story:** As an OrderAdmin user, I want to access the orders management page without feature toggle restrictions, so that I can always manage orders.

#### Acceptance Criteria

1. WHEN an OrderAdmin user navigates to the Orders_Page, THE System SHALL render the page without checking the `adminOrdersEnabled` Feature_Toggle
2. WHEN an OrderAdmin user is on the Orders_Page, THE System SHALL display the same order management functionality as a regular Admin user

### Requirement 6: Frontend Admin Dashboard Redirect for OrderAdmin

**User Story:** As an OrderAdmin user, I want to be redirected from the admin dashboard to the orders page, so that I am not shown admin features I cannot access.

#### Acceptance Criteria

1. WHEN an OrderAdmin user navigates to `/pages/admin/index`, THE System SHALL redirect the user to `/pages/admin/orders`
2. THE System SHALL perform the redirect before rendering the Admin_Dashboard content

### Requirement 7: Frontend Settings Page for OrderAdmin

**User Story:** As an OrderAdmin user, I want a simplified settings page with only relevant options, so that I am not confused by admin features I cannot use.

#### Acceptance Criteria

1. WHEN an OrderAdmin user opens the Settings_Page, THE System SHALL display the change password option
2. WHEN an OrderAdmin user opens the Settings_Page, THE System SHALL display the language switch option
3. WHEN an OrderAdmin user opens the Settings_Page, THE System SHALL display the theme switch option
4. WHEN an OrderAdmin user opens the Settings_Page, THE System SHALL display the logout option
5. WHEN an OrderAdmin user opens the Settings_Page, THE System SHALL NOT display the "admin panel" navigation entry

### Requirement 8: Frontend Navigation for OrderAdmin

**User Story:** As an OrderAdmin user, I want a minimal navigation with only order management and settings, so that I see only the features relevant to my role.

#### Acceptance Criteria

1. WHILE an OrderAdmin user is logged in, THE System SHALL display only two navigation entries: "订单管理" (Order Management) and "设置" (Settings)
2. WHILE an OrderAdmin user is logged in, THE System SHALL NOT display navigation entries for hub, mall, cart, profile, or other admin features

### Requirement 9: User Management Restrictions for OrderAdmin

**User Story:** As a SuperAdmin, I want to manage OrderAdmin users while preventing regular Admins from modifying them, so that OrderAdmin accounts are protected.

#### Acceptance Criteria

1. WHEN a regular Admin views the user list, THE System SHALL display OrderAdmin users in the list
2. WHEN a regular Admin attempts to edit, disable, or delete an OrderAdmin user, THE System SHALL return an error and prevent the operation
3. WHEN a SuperAdmin attempts to edit roles, disable, or delete an OrderAdmin user, THE System SHALL allow the operation
4. WHEN the batch points role selector is displayed, THE System SHALL NOT include OrderAdmin as a selectable target role
5. WHEN the claims role filter is displayed, THE System SHALL NOT include OrderAdmin as a selectable option

### Requirement 10: Role Exclusivity Enforcement

**User Story:** As a SuperAdmin, I want OrderAdmin to be mutually exclusive with all other roles, so that role boundaries remain clear and access control is predictable.

#### Acceptance Criteria

1. WHEN OrderAdmin is selected during role assignment, THE System SHALL clear all other assigned roles (Admin, SuperAdmin, Speaker, UserGroupLeader, Volunteer)
2. WHEN any other role is assigned to a user who currently has OrderAdmin, THE System SHALL remove OrderAdmin from the user's roles
3. WHEN the backend receives a role assignment request containing OrderAdmin alongside other roles, THE System SHALL reject the request with a validation error
4. THE System SHALL enforce role exclusivity in both the frontend role selector UI and the backend role assignment API

### Requirement 11: Internationalization for OrderAdmin

**User Story:** As a user of any supported language, I want to see the OrderAdmin role label in my language, so that the interface is fully localized.

#### Acceptance Criteria

1. THE System SHALL include a translation for the OrderAdmin role label in the Chinese (zh) Locale_File
2. THE System SHALL include a translation for the OrderAdmin role label in the English (en) Locale_File
3. THE System SHALL include a translation for the OrderAdmin role label in the Japanese (ja) Locale_File
4. THE System SHALL include a translation for the OrderAdmin role label in the Korean (ko) Locale_File
5. THE System SHALL include a translation for the OrderAdmin role label in the Traditional Chinese (zh-TW) Locale_File
6. THE System SHALL use the localized OrderAdmin label in all UI locations where the role name is displayed (role badges, invite selectors, user list)
